## Введение

Данный модуль посвящён продвинутым механизмам Android-разработки на языке Kotlin. В центре внимания — асинхронное программирование, фоновые вычисления и работа с системными API устройства. Каждое задание раскрывает конкретный инструмент платформы: от корутин и потоков данных до сервисов, будильников и аппаратных сенсоров.

Задания 1 и 2 выполнены в виде самостоятельных консольных (JVM) программ и не включены в состав Android-приложения. Задания 3–14 объединены в единое многофункциональное приложение **Module 4 App**, построенное на Jetpack Compose. Главный экран содержит список всех реализованных заданий, разбитых по тематическим разделам, и позволяет перейти к каждому из них одним нажатием.

Стек технологий приложения:
- **Kotlin 2.0** + **Jetpack Compose** (Material3, Navigation Compose)
- **Kotlin Coroutines** + **Kotlin Flow / StateFlow**
- **Android Services** (Foreground, Background, Bound)
- **WorkManager** (последовательные и параллельные цепочки)
- **FusedLocationProviderClient** + **Geocoder**
- **AlarmManager** + **BroadcastReceiver**
- **SensorManager** (акселерометр, магнитометр)
- **kotlinx.serialization**, **Coil**, **ViewModel + viewModelScope**

## Задания

### Задание 1 — Параллельная загрузка данных (консольное)

**Тема:** Kotlin Coroutines — `async` / `await`, `coroutineScope`

**Описание задачи:** Разработать консольную программу, которая параллельно загружает данные из нескольких источников (например, три разных «сервера», каждый с индивидуальной задержкой) и выводит сводный результат только после того, как все источники ответят.

**Решение:** Задача решена с помощью `coroutineScope { }` и функции `async { }`. Для каждого источника данных запускается отдельная корутина-`Deferred`. Вызов `awaitAll()` блокирует продолжение до получения всех результатов, после чего они агрегируются и выводятся. Суммарное время работы равно максимальной задержке среди источников, а не их сумме — что наглядно демонстрирует преимущество параллельного выполнения перед последовательным.

**Ключевые концепции:** структурный параллелизм, `async`/`Deferred`, `awaitAll`, время выполнения параллельных задач.

### Задание 2 — Поиск дублирующихся файлов (консольное)

**Тема:** Kotlin Coroutines — `Dispatchers.IO`, хэширование, отмена задач

**Описание задачи:** Написать консольную утилиту, которая рекурсивно обходит заданный каталог, вычисляет SHA-256 хэш каждого файла и выводит группы дубликатов (файлов с одинаковым содержимым).

**Решение:** Список файлов разбивается на пакеты, каждый пакет обрабатывается в отдельной корутине на `Dispatchers.IO`, что позволяет параллельно читать файлы с диска. Хэш вычисляется через `MessageDigest.getInstance("SHA-256")`. Результаты собираются в `ConcurrentHashMap<String, MutableList<Path>>`. Использование `supervisorScope` гарантирует, что ошибка чтения одного файла не прерывает обработку остальных. Итоговый вывод содержит только группы размером ≥ 2.

**Ключевые концепции:** `Dispatchers.IO`, `supervisorScope`, `ConcurrentHashMap`, SHA-256, параллельное чтение файлов.

### Задание 3 — Поиск репозиториев GitHub

**Тема:** Kotlin Coroutines — debounce через отмену `Job`, `StateFlow`, `LazyColumn`

**Описание задачи:** Экран с полем поиска, которое в реальном времени фильтрует список репозиториев GitHub. Поиск не должен запускаться на каждый введённый символ — только когда пользователь остановил ввод.

**Решение:** В `Task3ViewModel` хранится `Job` текущего поискового запроса. При каждом изменении строки предыдущий `Job` отменяется (`searchJob?.cancel()`), а новый стартует с `delay(500)` — это классическая реализация debounce без сторонних библиотек. По истечении паузы фильтрация выполняется на `Dispatchers.Default`, чтобы не блокировать главный поток. Результаты публикуются через `MutableStateFlow<SearchState>`, на который подписан UI. Список отображается через `LazyColumn` с карточками для каждого репозитория (название, описание, язык, звёзды). Данные загружаются из локального `assets/github_repos.json` (35 репозиториев).

**Ключевые концепции:** debounce через `cancel + delay`, `StateFlow`, `sealed class` для состояний UI, `Dispatchers.Default` для CPU-bound задач.

### Задание 4 — Социальная лента

**Тема:** Kotlin Coroutines — `async`/`await`, `supervisorScope`, параллельная загрузка данных карточек

**Описание задачи:** Лента публикаций, где для каждого поста параллельно подгружаются аватар пользователя и список комментариев. Ошибка загрузки одного поста не должна влиять на остальные.

**Решение:** В `Task4ViewModel` сначала последовательно загружаются посты и комментарии из JSON-файлов (`assets/social_posts.json`, `assets/comments.json`). Затем для каждого поста в `supervisorScope` запускается отдельная корутина, внутри которой `async { }` параллельно имитирует получение цвета аватара (задержка 500–1500 мс) и фильтрацию комментариев (300–1200 мс). После `await()` обоих результатов состояние поста переключается с `Loading` на `Ready`. Сбой в одной корутине изолируется `supervisorScope` — остальные посты продолжают загружаться. Каждый пост отображает цветной аватар с инициалами, заголовок, тело и раскрываемый список комментариев.

**Ключевые концепции:** `supervisorScope`, `async`/`await` внутри `supervisorScope`, изолированная обработка ошибок, пошаговое обновление `StateFlow`.

### Задание 5 — Счётчик времени (Foreground Service)

**Тема:** `Service`, `startForeground`, уведомления, `BroadcastReceiver` для связи с UI

**Описание задачи:** Таймер, который продолжает отсчёт даже после сворачивания приложения и отображается в шторке уведомлений с живым счётчиком.

**Решение:** `TimerForegroundService` запускается командой `startForegroundService()` и немедленно вызывает `startForeground()` с уведомлением в канале `CHANNEL_TIMER`. Внутри сервиса крутится корутина на `Dispatchers.Main`, которая каждую секунду инкрементирует счётчик, обновляет текст уведомления через `NotificationManagerCompat` и рассылает `broadcast` с action `BROADCAST_TICK` и текущим значением секунд. UI-экран регистрирует `BroadcastReceiver` через `DisposableEffect` и обновляет отображаемое время. При нажатии кнопки «Стоп» или при уходе с экрана сервис получает `ACTION_STOP` и вызывает `stopSelf()`. На Android 13+ запрашивается разрешение `POST_NOTIFICATIONS`.

**Ключевые концепции:** `startForeground`, `NotificationCompat.Builder`, обновление уведомления во время работы, связь сервис → UI через broadcast.

### Задание 6 — Одноразовый таймер (Background Service)

**Тема:** `Service`, одноразовое уведомление по истечении времени, `stopSelf()`

**Описание задачи:** Пользователь вводит количество секунд, нажимает кнопку — и через указанное время в шторке появляется уведомление, даже если приложение свёрнуто или закрыто.

**Решение:** `TimerBackgroundService` получает длительность через `Intent.extras` (`EXTRA_DURATION`). Внутри запускается корутина с `delay(duration * 1000L)`, после чего строится и публикуется `NotificationCompat` с каналом `CHANNEL_ALARM`. Сервис сам вызывает `stopSelf()` — то есть существует ровно столько, сколько нужно, не требуя явной остановки извне. UI после старта показывает индикатор ожидания и кнопку «Новый таймер» для сброса. Значение обрезается в диапазоне 1–3600 секунд.

**Ключевые концепции:** Background Service как одноразовый исполнитель, `stopSelf()` после выполнения задачи, жизненный цикл сервиса.

### Задание 7 — Случайное число (Bound Service)

**Тема:** `Service`, `IBinder`, `Binder`-паттерн, привязка и отвязка из Composable

**Описание задачи:** Сервис, который каждую секунду генерирует случайное число от 0 до 100. Экран подключается к сервису через привязку и отображает число в реальном времени.

**Решение:** `RandomNumberService` содержит вложенный класс `RandomBinder : Binder()`, метод `getService()` которого возвращает экземпляр сервиса. При вызове `onBind()` запускается корутина, публикующая `Random.nextInt(0, 101)` в `MutableStateFlow<Int>` каждую секунду. UI-экран (`Task7Screen`) использует `DisposableEffect` для управления `ServiceConnection`: при входе на экран вызывается `bindService()`, при уходе — `unbindService()`. После успешной привязки экран подписывается на `StateFlow<Int>` сервиса через `collectAsState()` и отображает текущее число с анимацией смены значений.

**Ключевые концепции:** `Binder`-паттерн, `ServiceConnection`, `bindService`/`unbindService`, `DisposableEffect` для управления жизненным циклом привязки в Compose.

### Задание 8 — Обработка фото (последовательная цепочка)

**Тема:** WorkManager — `beginWith().then().then()`, передача данных между Workers, прогресс

**Описание задачи:** Симуляция конвейерной обработки фотографии: сжатие → добавление водяного знака → загрузка в облако. Каждый этап выполняется отдельным Worker, результат предыдущего передаётся следующему.

**Решение:** В `PhotoWorkers.kt` реализованы три `CoroutineWorker`: `CompressPhotoWorker`, `WatermarkWorker`, `UploadWorker`. Каждый принимает имя файла через `inputData`, имитирует работу с задержкой, публикует прогресс (`setProgress(workDataOf(KEY_PROGRESS to n))`), и передаёт имя файла в `outputData`. Цепочка строится через `workManager.beginUniqueWork(..., compress).then(watermark).then(upload).enqueue()`. UI подписывается на `getWorkInfoByIdFlow()` каждого Worker и отображает суммарный `LinearProgressIndicator` (каждый этап — треть шкалы) и текстовый статус. По завершении `UploadWorker` выводится итоговое сообщение с именем файла.

**Ключевые концепции:** последовательная цепочка `WorkRequest`, `setProgress` / `WorkInfo.progress`, `ExistingWorkPolicy.REPLACE`, передача данных через `inputData`/`outputData`.

### Задание 9 — Прогноз погоды (параллельная обработка)

**Тема:** WorkManager — параллельные Workers, `beginWith(list)`, `ArrayCreatingInputMerger`, итоговый Worker

**Описание задачи:** Одновременная «загрузка» прогноза погоды для четырёх городов. После получения всех данных финальный Worker формирует сводный отчёт и показывает уведомление.

**Решение:** Для каждого города создаётся `OneTimeWorkRequest` на `WeatherWorker`, который принимает название города через `inputData` и генерирует случайную температуру. Все четыре запроса передаются в `workManager.beginWith(weatherRequests)`. Финальный `WeatherReportWorker` подключается через `.then(reportRequest)` с `setInputMerger(ArrayCreatingInputMerger::class)` — это позволяет получить массивы температур и городов из всех параллельных Workers. UI отображает карточку для каждого города с Live-статусом (`В очереди` → `Загружаем` → `+5°C`). По завершении в уведомлении и на экране появляется текстовый отчёт.

**Ключевые концепции:** параллельный запуск Workers списком, `ArrayCreatingInputMerger`, `addTag`, жизненные состояния `WorkInfo.State`.

### Задание 10 — Геолокация и адрес

**Тема:** `FusedLocationProviderClient`, `Geocoder`, обратное геокодирование, разрешения времени выполнения

**Описание задачи:** По нажатию кнопки получить текущие GPS-координаты устройства и преобразовать их в читаемый почтовый адрес (улица, город, страна).

**Решение:** Доступ к геолокации обёрнут в `suspendCancellableCoroutine<Location?>`: при отмене корутины вызывается `CancellationTokenSource.cancel()`, что прерывает запрос к `FusedLocationProviderClient.getCurrentLocation(PRIORITY_HIGH_ACCURACY, token)`. Обратное геокодирование реализовано с учётом версии API: на Android 13+ (`Build.VERSION_CODES.TIRAMISU`) используется асинхронный `Geocoder.GeocodeListener` внутри `suspendCancellableCoroutine`, на более старых — синхронный `getFromLocation()`. Функция `formatAddress()` собирает из объекта `Address` строку вида «ул. Ленина, 10 / Москва / Россия». При отсутствии разрешений кнопка инициирует запрос через `ActivityResultContracts.RequestMultiplePermissions`.

**Ключевые концепции:** `FusedLocationProviderClient`, `suspendCancellableCoroutine`, `CancellationTokenSource`, `Geocoder.GeocodeListener` (API 33+), версионирование API.

### Задание 11 — Напоминание о таблетке

**Тема:** `AlarmManager.setExactAndAllowWhileIdle`, `BroadcastReceiver`, `BOOT_COMPLETED`, `SharedPreferences`

**Описание задачи:** Ежедневное уведомление в 20:00 — «Пора принять таблетку». Напоминание должно восстанавливаться после перезагрузки устройства.

**Решение:** В `BootReceiver.scheduleAlarm()` вычисляется момент следующего 20:00: если текущее время уже за 20:00, берётся следующий день. Используется `AlarmManager.setExactAndAllowWhileIdle()` — единственный метод, надёжно срабатывающий в режиме Doze. В `AndroidManifest.xml` объявлен фильтр `BOOT_COMPLETED`, чтобы `BootReceiver` мог перепланировать будильник после перезагрузки. Состояние (включён/выключён, время следующего срабатывания) хранится в `SharedPreferences("alarm_prefs")`. `AlarmReceiver.onReceive()` строит уведомление и немедленно планирует следующее срабатывание через 24 часа. На Android 12+ экран проверяет `canScheduleExactAlarms()` и предлагает открыть системные настройки при необходимости.

**Ключевые концепции:** `setExactAndAllowWhileIdle`, Doze-режим, цепочечное перепланирование будильника, `BOOT_COMPLETED`, `SharedPreferences` для персистентности состояния.

### Задание 12 — Факты о животных (Cold Flow)

**Тема:** Cold Flow (`flow { }`), независимость каждого `collect()`

**Описание задачи:** По нажатию кнопки показывать случайный интересный факт о животном с имитацией загрузки. Каждое нажатие должно запускать поток данных заново.

**Решение:** `Task12ViewModel.getRandomFact()` возвращает `Flow<String>`, построенный через билдер `flow { delay(...); emit(...) }`. Это холодный поток — он не выполняет никакой работы, пока на него не подпишутся. При каждом нажатии кнопки в UI запускается новый `scope.launch { vm.getRandomFact().collect { ... } }`, что создаёт новый независимый запуск потока с отдельной задержкой (1.5–3 сек) и случайным фактом из списка 15 фраз. Время задержки варьируется, имитируя реальный сетевой запрос. UI использует `Crossfade` для плавного перехода между состояниями (загрузка / пустой экран / факт), что избавляет от конфликта с `ColumnScope.AnimatedVisibility`.

**Ключевые концепции:** холодный поток (`cold flow`), ленивый запуск, независимость подписок, `flow { }` builder, `emit`, задержка внутри потока.

### Задание 13 — Курс валют (StateFlow)

**Тема:** `MutableStateFlow`, `StateFlow`, автоматическое обновление через корутину, `collectAsState`

**Описание задачи:** Экран с живым отображением курса рубля к доллару, который автоматически обновляется каждые 5 секунд и показывает направление изменения.

**Решение:** `Task13ViewModel` содержит три `MutableStateFlow`: `rate` (текущий курс), `direction` (−1 / 0 / +1 для стрелки тренда) и `lastUpdate` (время последнего обновления). В `init` запускается корутина с `while(true) { delay(5000); updateRate() }`. Функция `updateRate()` генерирует новое значение в диапазоне 80–110 руб., округляет до копеек, вычисляет направление и обновляет все три потока атомарно. UI подписывается через `collectAsState()` и отрисовывает большую цифру курса, цветную стрелку (вверх/вниз) и строку с временем обновления. Кнопка «Обновить» вызывает `forceUpdate()` для немедленного внепланового обновления.

**Ключевые концепции:** `StateFlow` как горячий поток с кешированием последнего значения, `collectAsState()` в Compose, разница между `StateFlow` и `Flow`.

### Задание 14 — Компас

**Тема:** `SensorManager`, `TYPE_ACCELEROMETER`, `TYPE_MAGNETIC_FIELD`, `getRotationMatrix`, `getOrientation`

**Описание задачи:** Компас, стрелка которого указывает на север, используя аппаратные сенсоры устройства.

**Решение:** `Task14ViewModel` регистрирует единственный `SensorEventListener` сразу на два сенсора. При каждом событии данные копируются в буферы `accelerometerReading` и `magnetometerReading`. Функция `updateAzimuth()` вызывает `SensorManager.getRotationMatrix()` для построения матрицы поворота из обоих показаний, затем `SensorManager.getOrientation()` — для извлечения углов Эйлера. Первый элемент `orientationAngles[0]` — это азимут в радианах, который переводится в градусы (0°–360°). Для устранения дрожания применяется экспоненциальное сглаживание с коэффициентом 0.15. Результат публикуется в `MutableStateFlow<Float>`, на который подписан UI. Стрелка компаса поворачивается через `Modifier.rotate(-azimuth)`. Регистрация/разрегистрация сенсоров управляется через `DisposableEffect` и `LifecycleEventEffect` (RESUME/PAUSE), чтобы не тратить батарею в фоне.

**Ключевые концепции:** слияние показаний двух сенсоров, `getRotationMatrix`/`getOrientation`, радианы → градусы, экспоненциальное сглаживание, управление жизненным циклом сенсоров.

## Заключение

В ходе выполнения практических заданий модуля 4 были освоены ключевые механизмы Android-разработки, связанные с асинхронностью и фоновой работой:

**Корутины** (задания 1–4) продемонстрировали, как структурированный параллелизм позволяет писать читаемый асинхронный код без callback-ада. Паттерн debounce через отмену `Job`, изоляция ошибок через `supervisorScope` и параллельное накопление результатов через `async`/`awaitAll` — это практические инструменты, применимые в любом реальном приложении.

**Сервисы Android** (задания 5–7) показали три принципиально разных модели фоновой работы: Foreground Service для долгих операций с видимым уведомлением, Background Service для одноразовых задач «запустил и забыл» и Bound Service для двусторонней связи компонентов через `Binder`.

**WorkManager** (задания 8–9) оказался наиболее надёжным инструментом для отложенных задач с гарантией выполнения. Последовательные цепочки и параллельные пакеты Workers с объединением результатов через `ArrayCreatingInputMerger` покрывают большинство реальных сценариев.

**Системные API** (задания 10–11) раскрыли особенности работы с геолокацией и точными будильниками в условиях ограничений современного Android: Doze-режим, версионирование разрешений, необходимость восстановления состояния после перезагрузки.

**Flow** (задания 12–13) наглядно разграничили холодные и горячие потоки: `Flow` создаётся заново при каждой подписке и не хранит состояния, тогда как `StateFlow` всегда имеет актуальное значение и немедленно его отдаёт новым подписчикам. Это фундаментальное различие определяет, какой инструмент выбирать в зависимости от задачи.

**Сенсоры** (задание 14) показали, как объединить показания нескольких аппаратных датчиков для получения производного результата, и как грамотно управлять жизненным циклом подписки на сенсоры, чтобы не разряжать батарею.

Все задания интегрированы в единое приложение, что позволяет оценить их работу в реальном Android-окружении и проследить общие архитектурные решения: паттерн ViewModel + StateFlow, единый NavHost для навигации, централизованное управление каналами уведомлений.
